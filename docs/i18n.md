# Internationalization (i18n)

Arco UI strings are localized with **react-i18next**, following patterns from
OpenHands agent-canvas: a single source-of-truth JSON file, generated typed
keys, and ESLint to prevent new hardcoded copy in migrated surfaces.

Supported locales: **English**, **Español**, **Deutsch**, **日本語**, **简体中文**.
Users pick a language under **Settings → Appearance**. The choice is persisted
in `data/settings.json` (`Settings.locale`) and synced on boot via
`I18nLocaleSync`.

---

## Source of truth

| Path | Role |
| --- | --- |
| `src/i18n/translation.json` | All keys × all languages — edit this |
| `src/i18n/declaration.ts` | Generated `I18nKey` enum — do not edit manually |
| `public/locales/{lang}/arco.json` | Generated bundles loaded at runtime |
| `src/i18n/T.tsx` | `<T k={I18nKey.X} />` — locale-reactive JSX text |
| `src/i18n/index.ts` | i18next bootstrap, `applyArcoLocale()`, default export |

After editing `translation.json`, regenerate artifacts:

```bash
npm run make-i18n
```

`predev` and `prebuild` run this automatically.

## Developer patterns

### JSX text nodes

Prefer the typed wrapper (re-renders when locale changes):

```tsx
import { T } from "../../i18n/T";
import { I18nKey } from "../../i18n/declaration";

<T k={I18nKey.APPS$AUTOMATIONS_NEW} />
```

### String props (placeholders, aria-labels, titles passed as props)

Use `useTranslation().t()` in the component that owns the prop:

```tsx
const { t } = useTranslation();
<input placeholder={t(I18nKey.APPS$LIBRARY_SEARCH_APPS)} />
```

Avoid bare `i18n.t()` without a hook unless the parent already subscribes to
locale changes (e.g. via `useTranslation()` or a full-shell remount).

### Module headers and empty states

Shared dashboard components accept optional **key props** so copy stays reactive
without passing pre-translated strings:

```tsx
<ModuleHeader
  titleKey={I18nKey.OS$APP_AUTOMATIONS}
  subtitleKey={I18nKey.APPS$AUTOMATIONS_SCHEDULED_AND_EVENT_TRIGGERED_AGENT_RUNS_EACH_AUTOMATION}
/>
<EmptyState titleKey={I18nKey.APPS$AUTOMATIONS_NO_AUTOMATIONS_YET} />
<ModuleSection titleKey={I18nKey.APPS$AUTOMATIONS_RECOMMENDED} count={3} />
```

### System app window titles

Window titles stored in `windowStore` are captured at open time and can go stale
after a locale switch. **Display** code should resolve system-app titles at
render time:

```tsx
import { resolveWindowTitle } from "../os/resolveWindowTitle";

// WindowFrame, MobileShell, dock search, etc.
resolveWindowTitle(win); // system apps → systemAppTitle(id); others → win.title
```

Dock / palette open calls may still pass a snapshot title; that is fine for
internal storage — only the rendered label needs to be dynamic.

### Catalog and mock data

Preset cards, recommended lists, and stub nav must **not** hardcode user-facing
labels. Store `I18nKey` references; keep agent prompts and API payloads in
English when they are model input, not UI:

```ts
// src/apps/automations/catalog.ts
{
  id: "morning-briefing",
  nameKey: I18nKey.APPS$AUTOMATIONS_CATALOG_MORNING_BRIEFING_NAME,
  descriptionKey: I18nKey.APPS$AUTOMATIONS_CATALOG_MORNING_BRIEFING_DESC,
  prompt: "Create a morning briefing automation…", // agent-facing, stays EN
}
```

Render with `t(entry.nameKey)`. Search/filter should use localized labels, not
the English prompt text.

### Locale remount

`App.tsx` remounts `ElectronShell` with `key={i18n.language}` on locale change
so deeply nested trees pick up new copy. Prefer `useTranslation()` or `<T />`
anyway — remounting alone does not fix props computed once with `i18n.t()`.

### Agent-facing strings

Do **not** translate strings sent to the LLM (tool descriptions, automation
prompts, composer seeds). Only translate what the human sees.

## Adding a new string

1. Add an entry to `src/i18n/translation.json` with all five languages (or
   English first, then patch — see below).
2. Run `npm run make-i18n`.
3. Use `I18nKey.YOUR_KEY` in code via `<T />` or `t()`.
4. If the string is user-visible in a migrated file, ESLint
   (`i18next/no-literal-string`) should flag any remaining literals.

Key naming: `AREA$FEATURE_SNIPPET`, e.g. `APPS$AUTOMATIONS_RUN_NOW`,
`OS$APP_CHAT`, `COMMON$SAVE`, `INSTALL$WELCOME_TITLE`.

## Scripts

| Command | Description |
| --- | --- |
| `npm run make-i18n` | Regenerate `declaration.ts` and `public/locales/` |
| `npm run check-translation-completeness` | Fail on missing langs; **warn** on untranslated English copies |
| `npm run i18n:migrate` | Bulk JSX/prop migration pipeline (one-time / batch tooling) |
| `node scripts/i18n-patch-automations-locale.cjs` | Apply real es/de/ja/zh translations for Automations + window chrome |

Other one-off helpers live under `scripts/i18n-*.cjs` (import fixes, normalize,
stale-hook cleanup).

## Translation status (2026-07-07)

### Infrastructure — done

- react-i18next + http backend + browser language detector
- Settings locale picker and server persistence
- ESLint rule wired (`eslint-plugin-i18next`)
- Bulk migration of ~155 TSX files to `<T />` / `I18nKey`
- Dynamic window titles and localized window control aria-labels

### Content — in progress

The bulk migration created keys for most UI strings but **copied English into
every locale** as a placeholder. As of this writing:

- **~1,300 keys** still have identical English text in es/de/ja/zh-CN
- **`check-translation-completeness`** warns on these; CI is not yet strict
- **Automations** is the reference vertical slice: all `APPS$AUTOMATIONS_*`
  keys have real translations (see `i18n-patch-automations-locale.cjs`)

When localizing an app, follow the Automations pattern:

1. Wire UI through `I18nKey` (no hardcoded catalog/nav copy).
2. Use `titleKey` / `useTranslation()` for reactive props.
3. Add a patch script (or edit `translation.json` directly) with professional
   translations for that app's key prefix.
4. Run `npm run make-i18n` and verify in the browser (hard refresh after
   locale switch).

### Known gaps (not yet localized)

| Area | Notes |
| --- | --- |
| Settings stub nav | `settingsStubMock.ts` — hardcoded English section labels |
| Generator catalog | OpenUI component group names from upstream package |
| Podcast / music / video | Show titles and feed metadata (often external) |
| API-driven labels | Install checks, dynamic provider names |
| Server-side messages | Some API error strings returned to the client |

Tighten CI by flipping `check-translation-completeness` from warn to error once
placeholder debt is cleared (or allowlist brand/technical terms explicitly).

## Troubleshooting

**Mixed languages in one window** (e.g. Spanish title bar + Japanese body):
usually a stale bundle or props computed with `i18n.t()` before locale sync.
Hard refresh, switch language again, or ensure the component uses
`useTranslation()` / `titleKey`.

**Window title wrong language, body correct**: confirm `WindowFrame` uses
`resolveWindowTitle(win)`, not `win.title` alone.

**New keys show English in all locales**: expected until translations are added
to `translation.json` — the bulk migrate only creates keys, not copy.
